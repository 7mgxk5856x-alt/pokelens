# 手動・E2E テスト仕様書の作成 (完全自動実行モード)

**重要:** このワークフローは、ユーザーの介入なしに、開始から完了まで完全に自動で実行されるように設計されています。各ステップは完了後、ただちに次のステップへ移行してください。思考の途中でユーザーに確認を求めたり、作業を中断したりしないでください。**コミットは行いません。**

**用途:** `docs/product-requirements.md`（PRD）の受け入れ条件から、手動・E2E テスト仕様書 `docs/testing/e2e/manual-test-cases.md` を生成・更新する。コード由来の自動テスト一覧（`unit/` `integration/` の `automated-test-cases.md`）は `/add-feature`・`/fix-code` の担当であり、本コマンドの対象外。

**スコープ:** 本コマンドは **未カバーの受け入れ条件の追加（反映）専用**で、複数回の再実行を想定する（冪等な upsert）。既存の受け入れ条件が**変更・廃止**された場合の既存ケースの改訂・削除は自動では行わない。それらは手動で編集するか、`/review-test-cases` で陳腐化を検出して手で直す（「追加は自動・変更は人手」の方針）。

**引数（任意）:** 対象機能の指定（例: `/write-e2e-cases 機能3`）。省略時は PRD で優先度 P0 と記された機能全体を対象とする。

---

## ステップ1: 入力の把握

1. `docs/product-requirements.md` を読み、対象（引数で機能が指定されていればその機能、なければ **PRD で優先度 P0 と記された機能全体**）の**受け入れ条件**を抽出する。
  - **PRD が存在しない場合**: 「PRD がありません。先に `/setup-project` で作成してください」と案内して**終了する**。
2. 画面挙動・用語を正確に把握するため `docs/functional-design.md`・`docs/glossary.md` を参照する（初回生成時＝仕様書ファイルが無い場合は必ず参照。追記時は必要に応じて。存在しない場合はスキップし、その旨を記録する）。

## ステップ2: 既存仕様書の確認

1. `docs/testing/e2e/manual-test-cases.md` が存在するか確認する。
2. 存在する場合は、既存ケース（`MIT-` 連番・`対応要件` 列）を読み、**どの受け入れ条件が既にカバー済みか**を把握する。連番の最大値も控える。ファイルが空・破損などで `MIT-` 連番が読み取れない場合は、既存内容を保持しつつ連番を 1 から採番して扱う。

## ステップ3: 手動・E2E で検証すべき条件の抽出

受け入れ条件と既存ケースを突き合わせ、**未カバーの条件**を特定する。そのうち、手動・E2E（ブラウザ操作）で検証すべきものに絞る。

- 含める: UI コンポーネントの結合動作・画面表示・キーボード操作・データ準備パイプラインの実行確認など、自動テストで代替できない領域。
- 除外する: 純粋なロジック・計算（実数値・火力指数・素早さ計算・検索正規化など）は自動の単体テストが担保するため含めない。**単体テストの手動版は作らない**方針を守る。
- 除外したものは、その理由とともに完了報告に記す。

## ステップ4: ケースの生成・追記（冪等な upsert）

既存ファイルの体裁（次のとおり）に厳密に従う。

- 冒頭の宣言: 対象バージョン / 対象機能 / 実行環境 / 優先度 / 性能（スコープ宣言）
- 「表の項目」表、「サマリ」表
- セクションごとの前提条件 ＋ 表（ケースID `MIT-<連番>` / 種別 / 対応要件 / テスト観点 / 手順 / 期待結果 / 備考）

処理:

1. **ファイルが無い場合**: `docs/testing/e2e/` ディレクトリが無ければ作成し、上記フォーマットで新規作成して、対象の受け入れ条件から機能別セクションを起こしてケースを生成する。
2. **ファイルが有る場合**: 既存ファイルを丸ごと置き換えず、**未カバー分のケースのみ**該当セクション（または新セクション）に追記する（既存ケースは変更・削除しない）。`MIT-` 連番は既存最大値の続きから採番する（最大値が読み取れない場合はステップ2の定義に従い 1 から採番する）。
3. 各ケースに記入する項目:
  - 種別: 正常系 / 異常系 / 境界値
  - 対応要件: 検証対象の PRD 機能番号（例: `機能3`）
  - テスト観点・手順・期待結果: 受け入れ条件から具体的かつ**判定可能**に記述する（「正しく表示される」等の曖昧表現は使わない）
  - 備考: 後始末が必要なケースは「後処理: …」と明記する
4. サマリ表のセクション・ケース数・合計を更新する。

## ステップ5: 整合性チェック

1. `MIT-` 連番に重複・欠番がないか確認する。
2. `対応要件` が PRD の機能番号と一致しているか確認する。
3. 既存ケースを誤って変更・削除していないか確認する。
4. サマリ表のケース数合計が実際の行数と一致しているか確認する。

## ステップ6: 完了報告

1. 生成・追記したケース（ケースID と対応要件）を一覧で報告する。未カバー条件が 0 件で追記しなかった場合は「全受け入れ条件が既にカバー済みのため追記なし（ファイル変更なし）」と報告する。
2. 手動・E2E 対象外として**除外した受け入れ条件**を理由付きで報告する。
3. 生成物の点検として `/review-test-cases docs/testing/e2e/manual-test-cases.md` を次の手順として提示する（このコマンドからは自動実行しない）。

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で自動的に完了となる。

- ステップ3: 対象の受け入れ条件が、手動・E2E で検証すべきものとそうでないものに仕分けされている。
- ステップ4: 手動・E2E 対象の未カバー条件がすべて `docs/testing/e2e/manual-test-cases.md` にケースとして反映されている。
- ステップ5: 連番・対応要件・サマリ件数の整合性が取れている。整合性エラーを検出した場合は自律的に修正してから完了とする。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
