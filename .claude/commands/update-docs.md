# ドキュメント反映 (設計記述・事後反映の両モード)

**用途**: スペック駆動開発で **docs を設計の単一真実源** とするため、以下 2 モードを git 状態で自動切替する。

- **Phase 1 (PRD → 派生 docs への伝播 / pre-impl)** — *主たる役割*
  - **前提**: `docs/product-requirements.md`（PRD）に当該機能のセクション（ユーザーストーリー・受け入れ条件）が **既に書き込まれている** こと。PRD の編集は本コマンドの対象外（手動編集 or `/setup-project` で行う）。
  - PRD の該当セクションを **入力（仕様の出典）** として読み、派生 5 ドキュメント（機能設計書・アーキテクチャ・リポジトリ構造・用語集・開発ガイドライン）への **設計記述** をカテゴリ別ユーザー承認サイクルで進める。PRD 自体は本フェーズで書き換えない。
  - **`/add-feature` の前** に実行する。docs が固まることで後続の実装は docs を仕様として読むだけになり、AI の独断による設計逸脱を構造的に防ぐ。
- **Phase 3 (事後反映 / post-impl)**
  - 実装完了後の PRD 受け入れ条件チェックボックス（`[ ] → [x]`）更新・テスト一覧反映・docs 細部追従。
  - **`/add-feature` の後** に実行する。

**モード自動判定**: `git diff <base>...HEAD --name-only` の結果で判定する。
- 変更が **無し or `docs/` `.steering/` のみ** → Phase 1
- `src/` / `tests/` / `tools/` のいずれかに変更 **あり** → Phase 3
- 混在で判断が曖昧 → ユーザーに確認

**過去の失敗事例（再発防止）**: 機能 7（メガシンカ）で AI が `src/data/mega-evolutions.json` を独断配置した。原因は `/add-feature` 内で「設計と実装が連続実行」され、設計判断が一時ファイル（steering の design.md）に留まり永続ドキュメントに反映される前にコードが書かれたこと。**Phase 1 はこれを構造的に防ぐ**ためにある（docs が先・コードが後）。

**引数（任意）**:
- **Phase 1 を起動したいとき**: PRD の該当機能セクションを特定する **機能名** を渡す。AI は PRD 内を検索して当該セクションを読む。PRD で記述が不足している場合は **意図補足を続けて文字列で渡せる**（任意）。
  - 例 1: `/update-docs メガシンカ対応`
  - 例 2: `/update-docs メガシンカ対応 "C# パイプライン側で親 → メガ展開を行う schema 移行を想定"`（PRD の記述に不足があるとき）
- **Phase 3 を起動したいとき**: 引数なしで自動検出させるのが基本。曖昧時のみ機能名を渡す。
- **モードを明示したいとき**: 引数の先頭に `phase1:` または `phase3:` を付けて自動判定を上書きできる（git 差分がある状態で Phase 1 を強制したい等の例外時）。

**実行例**:
```bash
# Phase 1（PRD → 派生 docs 伝播）— PRD 編集後・/add-feature の前
> /update-docs メガシンカ対応

# Phase 3（事後反映）— /add-feature の後
> /update-docs
> /update-docs メガシンカ対応   # 機能名で文脈を絞りたい時

# モード強制（例外的）
> /update-docs phase1:メガシンカ対応
```

**重要（両モード共通）**:
- 各カテゴリのドキュメント変更は **必ずユーザーに案を提示してから適用**。一括書き換えしない。
- AI は **設計判断を独自に下さない**。既存パターン・docs と整合しない判断が必要なら、提案として明示しユーザーに承認 / 修正指示を求める。
- `git commit` は実行しない。

## ステップ1: モード判定とコンテキスト特定

1. **git 状態の取得**:
   - 現在のブランチ: `Bash('git branch --show-current')`
   - base 判定: ブランチ名から推測（`PN/feature/*` → `PN/release` / それ以外 → `main`）。存在しなければ `main` フォールバック
   - 変更ファイル: `Bash('git diff <base>...HEAD --name-only')`
2. **モード判定**:
   - 変更ファイルなし or `docs/` `.steering/` のみ変更 → **Phase 1**
   - `src/` / `tests/` / `tools/` 配下に変更あり → **Phase 3**
   - 混在で曖昧 → ユーザーに「Phase 1（設計記述）と Phase 3（事後反映）のどちらで進めますか？」と確認
   - 引数先頭に `phase1:` / `phase3:` がある場合は自動判定を上書き
3. **コンテキスト特定**:
   - Phase 1:
     - 引数の **機能名** で `docs/product-requirements.md`（PRD）内の該当セクションを検索する。**該当セクションが見つからない場合は実行を中断** し、「PRD に当該機能のセクションを先に書いてから本コマンドを実行してください（手動編集 or `/setup-project`）」と案内して終了する
     - PRD の該当セクション（ユーザーストーリー・受け入れ条件・優先度）を **仕様の出典** として読む
     - 引数末尾に意図補足の文字列が付いていれば、PRD の記述に対する追加文脈として取り込む
     - 関連する既存 docs（`docs/functional-design.md` / `docs/architecture.md` / `docs/repository-structure.md` / `docs/glossary.md` / `docs/development-guidelines.md`）を読み込み、既存パターンを把握する
   - Phase 3: ブランチ名から機能名を推測 → `.steering/` 配下から一致ディレクトリを探し、あれば `requirements.md` / `design.md` / `tasklist.md` を補助情報として読む（steering は `/add-feature` の内部成果物なので「読むだけ」）
4. **ユーザーへの初期提示**（5 行程度の要約）:
   - 検出モード（Phase 1 / Phase 3）と判定根拠（差分ファイル数等）
   - 機能名 or 意図要約
   - 後続ステップ（カテゴリ別の更新サイクル）に進む旨

## ステップ2 (Phase 1 のみ): 既存パターンの調査と設計提案の起草

> Phase 3 のときは本ステップをスキップして **ステップ4** へ。

PRD の該当セクションを **仕様の出典** として、派生 docs に書き込む設計判断を AI 自身で点検し、ユーザーに提示できる形で要約する。

1. **既存パターンの調査**:
   - 類似機能の配置（例: 手動メンテ JSON は `tools/PokelensTools/Patches/` 配下か）
   - 命名規則・型定義
   - 既存 docs（特に `docs/functional-design.md` / `docs/architecture.md` / `docs/repository-structure.md` / `docs/glossary.md` / `docs/development-guidelines.md`）に登録済みの用語・モジュール
2. **設計の主要選択肢の整理**:
   - PRD の受け入れ条件を満たすための新規ファイル/モジュール/API/データ schema/用語について、**配置と命名** を既存パターンとの一貫性を持つ案として **第一推奨** で提示する
   - 既存パターンに収まらない選択肢がある場合は、代替案として並べ、根拠と却下条件を明示
3. **ユーザーへの「設計提案サマリ」提示**:
   - functional-design / architecture / repository-structure / glossary / development-guidelines のうち、本機能で **触れるカテゴリ** を列挙（PRD は対象外。既に固定されているため）
   - 各カテゴリで何を追加 / 変更するかを 1〜2 行で要約
   - **AI が自信を持って決められない判断は質問として提示**（「`mega-stones.json` を `tools/PokelensTools/Patches/` に置く案でよいか？」等）
4. ユーザーから設計方針の確認・修正指示を受け、次のステップ3 でカテゴリ別承認に入る。

> **PRD が出典として不足する場合**: 受け入れ条件が曖昧で派生 docs の設計判断に十分な情報がない場合は、本コマンドの実行を中断し、「PRD の該当セクションを補強してから本コマンドを再実行してください」とユーザーに案内する。PRD の編集自体は本コマンドの対象外。

---

## ステップ3 (Phase 1 のみ): 派生 docs カテゴリ別の更新サイクル

> Phase 3 のときは本ステップをスキップして **ステップ4** へ。

以下 5 カテゴリ（PRD は対象外）を順に処理する。各カテゴリで「AI 提案 → ユーザー承認 → `Edit` 適用 → 次へ」を回す。

1. **機能設計書 (`docs/functional-design.md`)**:
   - 新規エンティティ・新規コンポーネント・新規 API・シーケンス図・ASCII レイアウトを追加
   - PRD の受け入れ条件と対応付けた「責務」「依存」「データ型」を明文化
2. **アーキテクチャ (`docs/architecture.md`)**:
   - 新規ファイル・新規ディレクトリをツリーに追記
   - 新規モジュールの依存関係（許可・禁止）を明記
3. **リポジトリ構造 (`docs/repository-structure.md`)**:
   - レイヤーへのファイル追加・新規ディレクトリの配置根拠を記載
4. **用語集 (`docs/glossary.md`)**:
   - 新規ドメイン用語・関連用語リンク・五十音目次（該当行）を追加
5. **開発ガイドライン (`docs/development-guidelines.md`)**:
   - 新規コーディング規約・配置ルール・命名規則を導入する場合のみ更新
   - 既存規約と矛盾がないかを必ず点検

各カテゴリでユーザーが「スキップ」を選んだ場合は、その旨を完了報告に明記してそのまま進める。

**完了時の合意**: 全カテゴリ処理後、ユーザーに「PRD の設計が派生 docs に伝播されました。`/add-feature <機能名>` で実装に進めます」と案内する。

---

## ステップ4 (Phase 3 のみ): PRD 受け入れ条件のチェック反映

> Phase 1 のときは本ステップをスキップして **ステップ7** へ。

1. `docs/product-requirements.md` の該当機能セクションを Read
2. 実装 + テストでカバーされた受け入れ条件 `- [ ]` を `- [x]` に切替候補として抽出
3. **対応根拠を 1 行で明示**（例: 「機能 7 `状態切替` → AET-039/040/042、`OwnPartyPanel#cycleMegaState` で達成」）
4. 対応するテストがない受け入れ条件は `- [ ]` のまま残し、**未カバー** として報告
5. PRD 本文の書き換え（例示の修正等）が必要なら **必ずユーザー承認** を得てから適用
6. 表面的なチェック切替のみは承認後に一括適用してよい

---

## ステップ5 (Phase 3 のみ): テストケース一覧の反映

対象:
- `docs/testing/unit/automated-test-cases.md`（フロントエンド Vitest・C# xUnit 単体）
- `docs/testing/integration/automated-test-cases.md`（C# 統合パイプライン）
- `docs/testing/e2e/automated-test-cases.md`（Playwright E2E）
- `docs/testing/e2e/manual-test-cases.md` は `/write-e2e-cases` の責務（**書き込まない**。未カバーがあれば案内）

**`<base>` の決定**: ブランチ名から `PN/release` / `main` を判定。`git rev-parse` で実在確認、不能ならユーザーへベースコミット指定を求める。

手順:
1. `git diff <base>...HEAD tests/` で追加・変更されたテストファイルとケース名を抽出
2. ID 接頭辞（`LD-` / `AET-` / `EIC-` 等）を現行接頭辞リストと突合
3. 各テスト一覧表に対する更新候補（行追加・サマリ件数・接頭辞）を提案
4. ユーザー承認後に `Edit`（新規作成は `Write`）で適用
5. `manual-test-cases.md` の `src/ui/` 関連カバレッジを点検し、未カバーがあれば `/write-e2e-cases <機能名>` を案内

> **テストコード自体は本コマンドの対象外**。`tests/**` の `.test.js` / `.spec.js` は `/add-feature`（実装ループ内）または `/fix-code`（ステップ4）が作成済みの前提。`git diff tests/` が空ならば本ステップをスキップして「テストコード未作成のため一覧反映なし」と報告する。

## ステップ6 (Phase 3 のみ): 永続ドキュメントの細部追従

Phase 1 で書いた設計と実装の細部に **軽微なズレ** がある場合の追従更新（例: テスト ID 反映、実数値の修正、文言の微調整、新規追加テストの引用）。

**重要（再発防止のための判定）**: 提案する変更が **Phase 1 承認の設計範囲を逸脱している** と感じたら、本ステップで適用してはならない。代わりにユーザーへ「実装が docs（Phase 1 承認内容）と乖離しています。設計の再記述が必要な可能性があるので、PRD の該当セクションを必要に応じて補強し `/update-docs <機能名>` で派生 docs を再伝播してから、必要に応じて `/add-feature` の再実行・`/fix-code` の修正を検討してください」と報告する。

軽微な追従に限って `Edit` で適用する。

---

## ステップ7 (両モード共通・任意): ステアリングへの反映

ステアリングディレクトリが見つかった場合のみ:
- Phase 1 完了時: `tasklist.md` の「設計記述（/update-docs Phase 1）完了」相当の項目（あれば）を `[x]`
- Phase 3 完了時: `tasklist.md` の「ドキュメント反映」項目を `[x]`、または振り返り欄に「/update-docs Phase 3 完了」を 1 行追記

steering を `/add-feature` 経由でしか使わない設計のため、本コマンドが新規に steering を作成・編集することはない（既存への局所追記のみ）。

---

## ステップ8: 完了報告

1. **変更したファイル一覧**を提示
2. **モード**（Phase 1 / Phase 3）と次のアクション案内:
   - **Phase 1 完了** → ユーザーへ「設計が docs に固定されました。`/add-feature <機能名>` で実装に進めます」と案内。**直接 `/add-feature` を起動しない**（ユーザーが docs 内容を再確認した上で明示的に呼び出すべきフェーズ移行のため）
   - **Phase 3 完了** → ユーザーへ「事後反映が完了しました。`/review-doc <パス>` / `/review-code` / `/suggest-commit-message` のいずれかへ」と案内
3. 未反映の項目（未カバーの受け入れ条件・テスト未作成・ユーザーがスキップを選んだカテゴリ等）を理由とともに報告

---

## 完了条件

- ステップ1 でモード判定が完了し、ユーザーに提示されている
- 該当 Phase の各カテゴリで、ユーザーから明示的な「承認」「適用」「OK」のいずれかが得られた変更のみが `Edit` / `Write` で適用されている
- ステップ7 該当時（ステアリング存在時）に既存セクションへの追記が完了している
- ステップ8 で変更ファイル一覧・モード・次のアクション案内が提示されている
- `git commit` は **実行していない**

> Phase 1 完了状態で `/add-feature` を AI が自動起動することは禁止。ユーザーが docs の最終内容を確認した上で次フェーズに移行する。
