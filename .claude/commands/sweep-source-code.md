# ソースコード一括レビュー & 自動修正 (`/sweep-source-code`)

**重要:** このワークフローはユーザーの介入を最小化し、自律的に実行されるよう設計されている。各ステップは完了後ただちに次のステップへ進む。ただし、対応方針の決定にユーザーの判断が必要な指摘は **修正せず残件として記録** し、コマンド完了時にまとめて報告すること。

**引数:** なし

**対象:** `src/**/*.js` と `tools/PokelensTools/**/*.cs`(テストプロジェクト `tools/PokelensTools.Tests/` は除外)

---

## 共通: 進行ログの出力ルール

ユーザーが進捗を画面上で把握できるよう、各ステップ・各ループのタイミングで進行ログをテキスト出力(=メインエージェントが画面に表示するテキスト応答)で必ず出すこと。Bash の `echo` ではなくテキスト応答として出力する(ツール出力に隠れず、ユーザーに直接表示される)。

### ステップ開始/完了ログ

各ステップを開始する直前と完了直後に以下のフォーマットで1行ずつ出力する:

```
▶ ステップN: <ステップ名> を開始
…(ステップ実行)…
✓ ステップN: <ステップ名> 完了
```

スキップした場合は `↷ ステップN: <ステップ名> をスキップ(理由: …)` の形にする。
中断する場合は `✗ ステップN: <ステップ名> を中断(理由: …)` の形にする。

### ステップ4 のループ進捗ログ

ステップ4 ではファイルごとのループを回すため、各イテレーションの**開始時**と**完了時**に進捗率を表示する。`i` は 1 始まり、`N` は対象ファイル総数、進捗率は完了済みファイル数を使って `floor(完了数 / N * 100)%` で算出する。

```
▶ [i/N] (XX%) <ファイルパス> のレビューを開始
…(4-1〜4-4 を実行)…
✓ [i/N] (XX%) <ファイルパス> 完了 (修正コミット Y 件 / 残件 Z 件)
```

ファイル内の指摘処理(4-4 の内部ループ)でも、各指摘の処理開始時に `  - 指摘 j/M: <タイトル>` のように 2 段インデントで出すと進捗が読みやすい(任意。指摘が多い大型ファイルでは推奨)。

## ステップ1: 事前チェックと作業ディレクトリ準備

**開始ログ:** `▶ ステップ1: 事前チェックと作業ディレクトリ準備 を開始`

1. 現在の git 状態を確認する:
   ```bash
   git status                    # 作業ツリーがクリーンか
   git rev-parse --abbrev-ref HEAD
   ```
   未コミットの変更がある場合は「未コミットの変更があります。コミット or stash 後に再実行してください」と報告して終了する。

2. 日付コンテキストを確立する:
   - 日付: `[現在の日付を YYYYMMDD 形式で取得]`
   - 作業ディレクトリ: `.steering/[日付]-sweep-source-code/`
   - 作業ブランチ名: `chore/sweep-source-code-[日付]`

3. 作業ディレクトリを作成する:
   ```bash
   mkdir -p .steering/[日付]-sweep-source-code/reports
   ```

4. 以下の空ファイルを作成する:
   - `.steering/[日付]-sweep-source-code/file-list.md` — 対象ファイル一覧 & 進捗
   - `.steering/[日付]-sweep-source-code/remaining.md` — ユーザー判断が必要な残件
   - `.steering/[日付]-sweep-source-code/summary.md` — 最終サマリー

## ステップ2: ブランチ作成

**開始ログ:** `▶ ステップ2: ブランチ作成 を開始`

```bash
git fetch origin
git checkout main
git pull --ff-only origin main
git checkout -b chore/sweep-source-code-[日付]
```

既に同名ブランチが存在する場合は、ユーザーに確認せず日付の末尾に `-2`, `-3` … を付与してユニークにする。

## ステップ3: レビュー対象ファイルの一覧化と順序決定

**開始ログ:** `▶ ステップ3: レビュー対象ファイルの一覧化と順序決定 を開始`
**完了ログ:** `✓ ステップ3: 対象ファイル N 件を確定` (N に実際の件数を入れる)

1. 対象ファイルを抽出する:
   ```bash
   find src -type f -name '*.js' | sort
   find tools/PokelensTools -type f -name '*.cs' \
     ! -path 'tools/PokelensTools/bin/*' \
     ! -path 'tools/PokelensTools/obj/*' | sort
   ```

2. 以下の優先順位でレビュー順を決定する(依存度の低いものから):
   - 第1群: `src/data/**/*.js`(純粋データ層)
   - 第2群: `src/logic/**/*.js`(純粋ロジック層)
   - 第3群: `src/ui/**/*.js`(UI 層)
   - 第4群: `src/main.js` などエントリ点
   - 第5群: `tools/PokelensTools/**/*.cs`(C# データ準備)

3. `file-list.md` に以下フォーマットで一覧を書き出す:
   ```markdown
   # レビュー対象ファイル一覧

   - [ ] src/data/foo.js
   - [ ] src/logic/bar.js
   ...
   ```

## ステップ4: 1ファイルずつレビュー・修正ループ

**開始ログ:** `▶ ステップ4: 1ファイルずつレビュー・修正ループ を開始 (対象 N 件)`

**`file-list.md` の全項目が `[x]` になるまで繰り返す。各ファイルは以下の手順で処理する。**

各ファイル処理の最初に進捗ログを出す:
```
▶ [i/N] (XX%) <ファイルパス> のレビューを開始
```
(`i` は処理中のファイルの順番、`N` は対象総数、`XX%` は `(i-1)/N * 100` の切り捨て=「これから i 件目に着手する時点の完了率」。i=1 のときは 0% から始まる)

### 4-1. レビュー実行

`Task` ツールで `code-reviewer` サブエージェントを起動する。**プロンプトは `/review-code`(引数ありモード)と完全に同一とし、追加の出力フォーマット指示は与えない。**

- `subagent_type`: `"code-reviewer"`
- `description`: `"Code detailed review"`
- `prompt`:
  ```
  [対象ファイル]を詳細にレビューしてください。

  以下の観点で評価してください:
  1. 可読性・命名: docs/development-guidelines.md の命名規則・コメント規約・フォーマットに準拠しているか
  2. 設計: 責務分離・純粋関数の原則・関数の長さ・import 依存ルール（src/logic は src/ui や src/data を import しない 等）
  3. テスト: ユニットテストの有無、エッジケースの網羅、命名規則
  4. セキュリティ: 入力検証、innerHTML 禁止、機密情報のハードコード
  5. 仕様整合性: docs/ の永続ドキュメント（PRD・機能設計書・アーキテクチャ）と矛盾していないか

  レビューレポートを作成してください。
  ```

サブエージェント本体(`.claude/agents/code-reviewer.md`)は上記5観点を含む**8観点**(可読性・命名 / 設計・責務分離 / テスト / セキュリティ / 仕様整合性 / ドメイン妥当性 / 解析性・例外処理 / 資源管理性)で評価し、`[必須]/[推奨]/[提案]` の3階層に整理されたレポートを返す。

### 4-2. レビュー結果の保存

サブエージェント出力をそのまま以下に保存する:

- 保存先: `.steering/[日付]-sweep-source-code/reports/[ファイルパスを `/` → `__` 置換した名前].md`
- 例: `src/logic/power-index-calc.js` → `reports/src__logic__power-index-calc.js.md`

### 4-3. 指摘事項の抽出と自動修正可否判定

メインエージェント自身が、保存済みのレビューレポートを読み込んで指摘を抽出する(サブエージェントには判定を求めない):

1. レポート中の `### 改善が必要な点` 配下の `[必須]` / `[推奨]` / `[提案]` セクションから各「問題」「提案」を上から順に列挙する。
2. それぞれを以下の基準で **自動修正可** / **要ユーザー判断** に分類する:
   - **自動修正可**: 修正案がレポートに具体的コードレベルで示されており、機械的な置換・追加・削除で完結し、副作用が局所に閉じる(import 順、変数名、コメント整理、未使用変数削除、ガード句追加、`innerHTML` → `textContent` 置換、明らかなテスト追加 等)
   - **要ユーザー判断**: 複数の設計選択肢がある / パブリック API のシグネチャを変える / 仕様(`docs/`)側か実装側のどちらを直すか分岐する / ドメイン知識でテストケースを選定する必要がある / 影響範囲が呼び出し側全体に及ぶ
3. 抽出結果を一時メモとして `.steering/[日付]-sweep-source-code/reports/[同じ名前]-issues.md` に書き出す(後の進捗確認に使用)。

### 4-4. 指摘事項の処理ループ

抽出した指摘について、上から順に1点ずつ以下を実施する:

#### 自動修正可能な指摘の場合

1. **修正適用:** `Edit` ツールで該当箇所を修正する。
2. **検証:**
   - JS ファイル修正時: `npm run lint` と `npm test` を実行
   - C# ファイル修正時: `dotnet build tools/PokelensTools` と `dotnet test tools/PokelensTools.Tests` を実行
   - 失敗時: 1度だけ修正の再調整を試みる。それでも失敗する場合は変更を破棄(`git restore <file>`)し、本指摘を **残件** に降格させる(理由: 自動修正で他の検証が失敗したため)。
3. **コミットメッセージ生成:**
   - 修正ファイルをステージする: `git add <file>`
   - 既存の一時ファイルをクリーンアップする: `rm -f .claude/tmp/commit-msg-*.txt && mkdir -p .claude/tmp`
   - `Task` ツールで `commit-message-writer` サブエージェントを起動:
     - `subagent_type`: `"commit-message-writer"`
     - `description`: `"Draft commit message"`
     - `prompt`: `/suggest-commit-message` のステップ2と同じテンプレートを使用(差分は `git diff --staged` を渡す)
   - 返却されたメッセージを `.claude/tmp/commit-msg-N.txt`(N は 1 から始まる連番)に書き出す。**サブエージェントが分割を推奨した場合は推奨数ぶんのファイルを作成する。**
4. **コミット実行:**
   - **単一コミットの場合(分割提案なし):**
     ```bash
     git commit -F .claude/tmp/commit-msg-1.txt
     ```
   - **分割推奨の場合:** サブエージェントが提示した分割案の通りに、コミット単位ごとに以下を繰り返す:
     1. 該当コミットに含めるファイル/ハンクのみをステージし直す:
        - ファイル単位の分割: 一度 `git reset HEAD` でアンステージし、対象ファイルだけを `git add <file>` する
        - ハンク単位の分割: `git reset HEAD <file>` でアンステージしてから `git add -p <file>` 相当の分割を行う(対話入力できない環境なので、`git apply --cached` で個別パッチを当てるなど非対話手段を用いる)
     2. 該当する `.claude/tmp/commit-msg-N.txt` でコミットする:
        ```bash
        git commit -F .claude/tmp/commit-msg-N.txt
        ```
     3. 次のコミットへ進む(N をインクリメント)
   - すべてのコミットが終わったら作業ツリーがクリーンであることを `git status` で確認する。残ステージがある場合は、その分も追加コミットして取りこぼしを防ぐ。
5. **進捗記録:** `file-list.md` 内の該当指摘番号に対し、生成したコミット SHA(分割時は複数)を脚注として記録する(任意。最低限ファイルチェックは 4-5 で行う)。

#### ユーザー判断が必要な指摘の場合

修正は行わず、`remaining.md` に追記する:

```markdown
## [ファイルパス] - 指摘#[番号]: [タイトル]

- 観点: [観点番号]
- 優先度: [優先度]
- 場所: [ファイルパス:行番号]
- 問題: [問題の説明]
- 推奨修正案: [サブエージェントが出した案]
- ユーザー判断が必要な理由: [設計判断 / 仕様確認 / 影響範囲広い 等]
```

「ユーザー判断が必要」と扱う典型例:
- 関数の責務分割など、複数の設計選択肢があり、どれを取るかで以降の構造が変わるもの
- 外部仕様(`docs/`)との不整合だが、コード側を直すか docs 側を直すかで意味が変わるもの
- 新規テストの追加で、テストケース選定にドメイン知識が必要なもの
- パブリック API のシグネチャを変更するもの(呼び出し側全体への影響あり)

### 4-5. ファイル単位の完了処理

そのファイルの全指摘の処理が終わったら:
1. `file-list.md` の該当エントリを `[ ]` → `[x]` に更新する。
2. 進捗完了ログを出力する:
   ```
   ✓ [i/N] (XX%) <ファイルパス> 完了 (修正コミット Y 件 / 残件 Z 件)
   ```
   `XX%` は `i/N * 100` の切り捨て(=この時点で完了したファイル数の割合)。

### 4-6. ループ継続

`file-list.md` に未完了(`[ ]`)があれば次のファイルへ進む。すべて `[x]` になったらステップ5へ。

ステップ4 全体の完了ログを出力する: `✓ ステップ4: 1ファイルずつレビュー・修正ループ 完了 (全 N 件処理)`

---

### ※ ループ内の例外処理ルール ※

- **指摘が0件のファイル:** そのファイルを `[x]` に変えるだけでコミットせずに次へ進む。
- **指摘の中身が重複している場合:** 最初の1件のみ処理し、残りは `remaining.md` に「重複指摘」として記載しスキップする。
- **検証 (lint/test) が初回から失敗 (修正以前の問題) する場合:** ユーザーに「既に lint/test が失敗しています」と報告して中断する。

## ステップ5: 最終検証

**開始ログ:** `▶ ステップ5: 最終検証 を開始`
**完了ログ:** `✓ ステップ5: 最終検証 完了 (lint/test/build/dotnet test 全パス)`

全ファイル処理完了後、以下を実行して全体が壊れていないことを最終確認する:

```bash
npm run lint
npm test
dotnet build tools/PokelensTools
dotnet test tools/PokelensTools.Tests
```

いずれかが失敗した場合は問題を特定し、追加コミットで修正する。修正不可能な場合は中断してユーザーに報告する。

`.steering/[日付]-sweep-source-code/summary.md` に以下を書き出す:

## ステップ7: プッシュとプルリクエスト

**開始ログ:** `▶ ステップ7: プッシュとプルリクエスト を開始`

**残件の有無で分岐する。** 分岐結果に応じて完了ログを出し分ける(7-a なら `✗ ステップ7: 残件 Z 件あり、プッシュ/PR をスキップ`、7-b なら `✓ ステップ7: プッシュ & PR 作成完了`)。

### 7-a. 残件がある場合(`remaining.md` に1件以上記載がある場合)

ここで処理を**中断**し、リモートへのプッシュおよびプルリクエスト作成は**行わない**。

理由: 残件はユーザー判断を要する未解決の指摘であり、それを抱えたまま PR を出すと、レビュアーが何を判断すべきか不明確になり、また残件解消後にコミット履歴を綺麗に追加するのが難しくなる。

このケースでは、ステップ8(最終報告)で以下を明示してユーザーに引き継ぐ:
- 残件一覧と `remaining.md` のパス
- ローカルにコミット済みであること(ブランチ名)
- ユーザーが残件を解消したあとに手動で `git push -u origin <ブランチ名>` および `gh pr create` を実行する手順、または再度 `/sweep-source-code` を残件解消後に再実行する選択肢

### 7-b. 残件がない場合(`remaining.md` が空 / `Z = 0`)

1. リモートへプッシュ:
   ```bash
   git push -u origin chore/sweep-source-code-[日付]
   ```

2. プルリクエスト作成(`gh` 利用):
   ```bash
   gh pr create --base main --title "chore: ソースコード一括レビュー & 自動修正 ([日付])" --body "$(cat <<'EOF'
   ## 概要
   `/sweep-source-code` によるソースコード一括レビュー & 自動修正の結果。

   ## 対象
   - src/**/*.js
   - tools/PokelensTools/**/*.cs

   ## 結果サマリー
   - 自動修正コミット数: [Y]
   - 残件(要ユーザー判断): 0

   ## 詳細
   - サマリー: `.steering/[日付]-sweep-source-code/summary.md`
   - レビューレポート: `.steering/[日付]-sweep-source-code/reports/`

   ## レビュー観点
   `/review-code` と同一(可読性・設計・テスト・セキュリティ・仕様整合性)。

   🤖 Generated with [Claude Code](https://claude.com/claude-code)
   EOF
   )"
   ```

3. PR URL を控える(最終報告で提示する)。

4. **main ブランチへ復帰する:** PR 作成が成功した時点でフィーチャブランチでの作業は終了したため、元の main へ戻す。
   ```bash
   git checkout main
   ```
   作業ツリーに残ステージや変更がある場合は、まず `git status` で確認し、不要な変更であれば破棄、必要な変更であればフィーチャブランチに追加コミットしてから main に切り替えること(`-f` は使わない)。
   作成済みのフィーチャブランチ `chore/sweep-source-code-[日付]` はローカル・リモート両方に残しておく(PR がマージされるまで削除しない)。

## ステップ8: 最終報告

**開始ログ:** `▶ ステップ8: 最終報告 を開始`
**完了ログ:** `✓ ステップ8: 最終報告 完了 — /sweep-source-code 全工程終了`

ステップ7の分岐に応じて、以下いずれかのフォーマットでユーザーに最終結果を提示する。

### 8-a. 残件がある場合(ステップ7-a 経由)

```markdown
# /sweep-source-code 中断(残件あり)

ローカルでの自動修正コミットは完了しましたが、ユーザー判断を要する残件があるため、
**プッシュおよびプルリクエストの作成は行いませんでした。**

## ブランチ
chore/sweep-source-code-[日付] (ローカルのみ、未プッシュ)

## 統計
- 対象ファイル: [N] 件
- 検出指摘: [X] 件
- 自動修正コミット: [Y] 件
- 残件: [Z] 件

## 残件一覧(ユーザー判断要)
1. [ファイル] - [タイトル] (優先度: 高/中/低)
2. ...

## 次のアクション

## 完了条件

以下のいずれかを満たした時点で完了とする(残件の有無で分岐する)。

### 残件なしの完了条件

- 対象ファイルのレビューレポートが全て `.steering/[日付]-sweep-source-code/reports/` 配下に保存されている
- 自動修正可能な指摘がすべて個別コミットされている
- `remaining.md` が空(残件 0 件)である
- `npm run lint` / `npm test` / `dotnet build` / `dotnet test` がすべて成功している
- `chore/sweep-source-code-[日付]` ブランチが origin にプッシュされている
- main 宛 PR が作成されている
- カレントブランチが main に戻されている

### 残件ありで中断する場合の完了条件

- 対象ファイルのレビューレポートが全て `.steering/[日付]-sweep-source-code/reports/` 配下に保存されている
- 自動修正可能な指摘がすべて個別コミットされている
- ユーザー判断が必要な指摘がすべて `remaining.md` に記録されている
- `npm run lint` / `npm test` / `dotnet build` / `dotnet test` がすべて成功している
- 残件があるため**プッシュおよび PR 作成はスキップ**し、ステップ8-a の形式でユーザーに残件と次のアクションを報告している

## 注意事項

- 1ファイルあたり数分〜十数分かかる場合がある。対象ファイル数が多い場合は十分な実行時間を確保すること。
- 各サブエージェントは独立コンテキストのため、メインのコンテキストを節約できる。
- 修正のたびに lint/test を実行するため、途中で他の修正が誤って壊れることを早期検知できる。
- `commit-message-writer` がコミットの分割を提案した場合は、必ずその提案通りに分割してコミットする(1指摘に対して複数コミットになる可能性がある)。これにより1コミット = 1論理単位の原則を保つ。
- `git push --force` は禁止。同名ブランチが既存の場合はブランチ名末尾でユニーク化する(ステップ2参照)。
- pre-commit フックをスキップする `--no-verify` は使用しない。フックが落ちた場合は根本原因を解消する。
- `.claude/tmp/commit-msg-*.txt` は再実行ごとに上書きされる一時領域(`.gitignore` 済み)。
