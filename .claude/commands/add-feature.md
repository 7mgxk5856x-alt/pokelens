# 新機能の実装 (docs を仕様とする実装専念モード)

**重要:** 本コマンドは **設計フェーズを含まない**。設計は前段の `/update-docs` (Phase 1) で永続ドキュメント（PRD・機能設計・アーキテクチャ・リポジトリ構造・用語集・開発ガイドライン）に固定された前提で、その docs を仕様として読み実装に専念する。

- `/update-docs` Phase 1 を **先に** 実行して設計を docs に書き込む
- 本コマンド `/add-feature` は docs を読んで実装計画に分解し、test/lint を通すまでを自動化する
- 実装中に docs に書かれていない判断が必要になったら **必ず中断してユーザーに報告**（後述ステップ5 のルール D）

**過去の失敗事例（再発防止）**: 機能 7（メガシンカ）で AI が `src/data/mega-evolutions.json` を独断配置した。原因は本コマンドが「設計 → 実装」を一気通貫で実行していたこと。今回の改修で **設計は `/update-docs` (Phase 1) に切り出し、本コマンドは「docs に従って実装する」だけ** に縮退した。

**引数:** 機能名（例: `/add-feature メガシンカ対応`）。docs 上で当該機能セクションが特定できる必要がある。

**前提チェック**: 起動時に以下を満たさない場合は実装に進まない:
- `docs/product-requirements.md` に該当機能セクションが存在する
- 機能設計書・アーキテクチャ・用語集等、影響が及ぶ永続ドキュメントが Phase 1 で更新済みである（AI から見た「読むべき仕様」が揃っている）

引数省略時は機能名の入力をユーザーに求める。

**ドキュメント反映は本コマンドの対象外**: 実装完了後は `/update-docs`（Phase 3 自動検出モード）で PRD チェックボックス更新・テスト一覧反映を行う。

---

## ステップ1: 準備と前提確認

1. タスクコンテキストを確立:
   - 機能名: `[引数で与えられた機能名]`
   - 日付: `[YYYYMMDD]`
   - ステアリングディレクトリパス: `.steering/[日付]-[機能名]/`
2. **前提チェック**:
   - `docs/product-requirements.md` に当該機能セクションが存在するか確認
   - **存在しない場合は実装に進まず**、「先に `/update-docs "<機能の意図文>"` で設計を docs に書き込んでください」とユーザーに案内して終了
3. ステアリングディレクトリと空ファイル（`requirements.md` / `design.md` / `tasklist.md`）を作成

---

## ステップ2: docs（仕様の単一真実源）を読む

1. `CLAUDE.md` でプロジェクト全体を把握
2. **必ず読むべき docs（仕様情報源として）**:
   - `docs/product-requirements.md` の当該機能セクション全文
   - `docs/functional-design.md` の関連エンティティ・コンポーネント・API 定義
   - `docs/architecture.md` の関連ファイル・依存関係
   - `docs/repository-structure.md` の関連レイヤー配置
   - `docs/glossary.md` の本機能で参照される用語
   - `docs/development-guidelines.md` の関連規約
3. 読んだ内容から「実装が満たすべき仕様」を抽出する（docs の言葉をそのまま使う）

---

## ステップ3: 既存パターンの確認

1. `Grep` で機能関連キーワードを `src/` と `tools/` から検索
2. docs に書かれた既存パターン（配置ルール・命名・型定義）と一致する実装を特定
3. **新規ファイル / 新規モジュールを足す必要があれば、docs に既に書かれている前提**。書かれていないなら本コマンドではなく `/update-docs` (Phase 1) に戻る

---

## ステップ4: 実装計画フェーズ（ステアリングファイル生成）

1. `Skill('steering')` を **モード1（ステアリングファイル作成）** で実行する
2. **生成内容の制約**:
   - `requirements.md`: docs の受け入れ条件を **引用** する（独自に書き直さない。`docs/product-requirements.md` 機能 X の項を参照、と記述）
   - `design.md`: **実装計画** に縮退する。新たな設計判断を書かない。次のセクションのみ含める:
     - **影響ファイル一覧**: docs（特に architecture.md / repository-structure.md）に既に書かれた配置に従って、変更/新規ファイルを列挙
     - **実装手順の分解**: docs の API 定義・データ型・責務をどの順で実装するか
     - **テスト計画**: 追加する単体テスト（接頭辞 + 観点）/ E2E テスト（AET 連番 + 観点）。観点は受け入れ条件に対応付ける
   - `tasklist.md`: 実装タスクとテスト追加タスクを **必ず両方** 並列で列挙
3. **このステップ完了後、ステップ4.5 へ進む（実装に進まない）**

---

## ステップ4.5: 実装計画の整合性チェック（ユーザー確認）

**設計判断のレビューではなく「実装計画が docs と一致しているか」のチェック** を行う。設計の中身そのものは Phase 1 で承認済みだから、ここでは深掘りしない。

1. `design.md` から **影響ファイル一覧** と **テスト計画** をユーザーに要約提示する
2. ユーザーは以下のいずれかを判定:
   - **そのまま実装に進む**: docs と整合している → ステップ5 へ
   - **計画を修正したい**: 軽微な順序変更や粒度調整に限定。設計の追加・変更は docs に書かれていない以上ここで決めない
   - **設計が docs に書かれていない**: 中断し、ユーザーに `/update-docs phase1:"..."` で docs を更新してから本コマンドを再実行するよう案内
3. AI 自己点検観点:
   - 影響ファイル一覧の配置が `docs/architecture.md` / `docs/repository-structure.md` の記述と一致しているか
   - テスト計画が `docs/product-requirements.md` の受け入れ条件をカバーしているか
   - **docs に未記載の判断（ファイル名・schema・新規 API シグネチャ等）が `design.md` に紛れ込んでいないか**

---

## ステップ5: 実装ループ (tasklist.md の完全消化)

**ループ開始**: `tasklist.md` の全タスクが `[x]` になるまで自動で繰り返す。

1. `tasklist.md` を Read
2. 未完了タスク `[ ]` がなければ → ステップ6 へ
3. 未完了タスクがあれば:
   - 先頭の未完了タスクを 1 つ特定
   - 必要な実装を実行（`Skill('steering')` モード2 / `Skill('development-guidelines')` 規約遵守）
4. 完了したタスクを `[ ]` → `[x]` に更新
5. ループ継続

---
### ※ 実装ループ内の例外処理ルール ※

- **ルールA: タスクが大きすぎる場合**: サブタスクに分割し、`Edit` で挿入してループ継続
- **ルールB: 技術的理由でタスクが不要になった場合**: `[x] ~~タスク名~~ (理由: ...)` の形式で完了マーク、ループ継続
- **ルールC: 実装計画レベルの軽微な調整が必要**: タスクの順序・粒度・命名の微修正なら自律的に行いループ継続
- **ルールD (重要): docs に書かれていない設計判断が必要になった場合**
  - **絶対に自分で判断しない**。ループを中断する
  - 「docs に未記載の判断が必要になりました: [具体的に何が]」とユーザーに報告
  - ユーザーは `/update-docs phase1:"<追加すべき設計意図>"` で docs を更新したのち、本コマンドを再開する想定
  - 例: 「新規ファイルが必要だが配置先が docs/architecture.md に書かれていない」「データ schema の細部が functional-design.md で未定義」「新規 API のシグネチャの一部が機能設計書で曖昧」
- **❌ 絶対禁止**:
  - 未完了タスクの意図的スキップ・放置
  - **docs に書かれていない判断を独断で実装に反映すること（ルールD を回避する目的での「軽微な調整」名目の設計判断も含む）**

---

## ステップ6: 自動テストの実行

1. `tasklist.md` 全タスクが完了したことを最終確認
2. 以下を順に実行:
   ```bash
   Bash('npm test')
   Bash('npm run lint')
   ```
3. エラーがあれば原因分析・修正コード適用 → 再実行。**ただし「docs に書かれていない設計判断を伴う修正」が必要ならルールD で中断**

---

## ステップ7: 実装検証（サブエージェント起動）

1. `Task` ツールで `implementation-validator` サブエージェントを起動:
   - `subagent_type`: "implementation-validator"
   - `description`: "Implementation quality validation"
   - `prompt`: "今回実装した `[機能名]` の品質を検証してください。対象ファイルは `[実装したファイルのパスリスト]` です。コーディング規約、エラーハンドリング、テスト可能性、既存パターンとの整合性、**そして docs（PRD・機能設計・アーキテクチャ等）に書かれた仕様との整合性** を重点的に確認してください。"
2. 重大な指摘があれば修正を適用（ルールD 違反がないか特に注意）

---

## ステップ8: 振り返りとブランチ確認

1. `Skill('steering')` モード3 で `tasklist.md` に振り返りを記録（完了日・計画と実績の差分・学び・改善提案）
2. ブランチ確認: `Bash('git branch --show-current')`
   - `main` 上だった場合のみ完了報告末尾に注意を含める

---

## ステップ9: 後続フェーズの案内

完了報告に以下を含める:

> 📝 ドキュメント反映は本コマンドのスコープ外です。続けて **`/update-docs`** を実行することで、PRD のチェックボックス更新・テストケース一覧反映などを進められます（git 差分が `src/` / `tests/` に及んでいるため、Phase 3 モードで自動起動します）。

---

## 完了条件

- **ステップ1 前提チェック**: 当該機能の docs セクションが存在することが確認されている
- **ステップ4.5**: 実装計画が docs と整合していることをユーザーが承認している
- **ステップ5**: `tasklist.md` の全タスクが完了（`[x]` または正当な理由でスキップ）
- **ステップ5 ルールD 遵守**: docs 未記載の設計判断は全てユーザーへエスカレーションされ、AI が独断で適用していない
- **ステップ6**: `npm test`・`npm run lint` がエラーなく成功
- **ステップ7**: `implementation-validator` の検証をパス
- **ステップ8**: 振り返り記載・ブランチ確認の付記が完了
- **ステップ9**: `/update-docs` (Phase 3) の案内が完了報告に含まれている

ルールD で中断した場合は「ユーザー判断待ち」として正常な中断であり、完了とはみなさない。
