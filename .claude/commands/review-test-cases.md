# テストケースレビュー

引数: テストケース仕様書のパス(例: `/review-test-cases docs/testing/e2e/manual-test-cases.md`)

`test-case-reviewer` サブエージェントを使い、**テスト設計技法**と**テスト観点**の2軸で
テストケース仕様書をレビューする。PRD の受け入れ条件と突き合わせ、観点漏れ・不足ケース・
記述品質の問題を指摘する。**テストケースの自動修正は行わない。**

> テストコードそのものの実装レビューは `/review-code` を使用する。
> 本コマンドは「テストケースとして十分か・正しく書けているか」を評価する。

## 実行方法

```bash
claude
> /review-test-cases docs/testing/e2e/manual-test-cases.md
```

主な対象:

- 手動（E2E）テスト仕様書: `docs/testing/e2e/*.md`
- 自動テストのケース一覧: `docs/testing/unit/*.md` / `docs/testing/integration/*.md`

## 手順

### ステップ1: 対象の存在確認

指定されたテストケース仕様書が存在するか確認する。
引数が省略された場合は、`docs/testing/` 配下のテストケース仕様書一覧を提示し、
対象パスの指定を促して終了する。

### ステップ2: test-case-reviewer サブエージェント起動

Task tool を使用して test-case-reviewer サブエージェントを起動する:

- subagent_type: "test-case-reviewer"
- description: "Test case review"
- prompt: "[対象パス] のテストケース仕様書を詳細にレビューしてください。\n\n## レビュー方針\n- docs/product-requirements.md（PRD）の受け入れ条件と各ケースを突き合わせ、要件カバレッジを評価する\n- 必要に応じて docs/functional-design.md・docs/architecture.md も参照する\n\n## 評価の2軸\n1. テスト設計技法（網羅性）: 同値分割／ドメイン分析・境界値分析・デシジョンテーブル・状態遷移テスト・組み合わせ／ペアワイズ・ユースケース／シナリオ・エラー推測。機能の性質に向いた技法が使われ、観点漏れがないか\n2. テスト観点: カバレッジ（要件適合性・網羅性・境界値・異常系・入力妥当性・出力妥当性・データ整合性）／品質特性（UI整合性・性能・セキュリティ・互換性）／記述品質（期待結果明確性・前提条件・後処理・再現性・独立性・可読性・保守性・優先度）\n\n## 成果物\n- 要件カバレッジ表（PRD条件↔ケースID↔状態）\n- 優先度別の改善点（[必須]/[推奨]/[提案]）\n- 追加すべきテストケース案（既存表フォーマットに合わせる）\n\nレビューレポートを作成してください。自動修正は行わないでください。"

### ステップ3: レビュー結果の要約

サブエージェントが作成したレビューレポートの要点を抽出し、ユーザーに報告する。
特に「未カバーの受け入れ条件」と「追加すべきテストケース案」を明示する。

## 出力形式

```markdown
# テストケースレビュー結果

## 対象: [ファイル名]

### 主な改善点

1. [改善点1] (優先度: 高/中/低)
2. [改善点2] (優先度: 高/中/低)
3. [改善点3] (優先度: 高/中/低)

### 未カバーの受け入れ条件

- [機能番号・条件] — [対応案]

### 追加推奨ケース（抜粋）

- [ケースID案]: [テスト観点] / [期待結果の要点]

### 総合評価

[1-5]/5

### 次のアクション

- [推奨対応1]
- [推奨対応2]

詳細なレポートはサブエージェントの出力を参照してください。
```

## 注意事項

- レビューは詳細な分析のため、数分かかる場合があります
- サブエージェントは独立したコンテキストで動作するため、メインエージェントのコンテキストは消費しません
- 自動修正は行いません。検出された問題の修正・ケース追加は別途実施してください
- 設計技法・観点は重複を精査のうえ集約しています（詳細は `.claude/agents/test-case-reviewer.md` を参照）
