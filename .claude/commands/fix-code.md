# ソースコードの修正 (完全自動実行モード)

**重要:** このワークフローは、ユーザーの介入なしに、開始から完了まで完全に自動で実行されるように設計されています。各ステップは完了後、ただちに次のステップへ移行してください。思考の途中でユーザーに確認を求めたり、作業を中断したりしないでください。

**用途:** 既存ソースコードの修正（バグ修正・リファクタ・`/review-code` の指摘反映など）を、対応する自動テストの追加・変更と `docs/testing/` の一覧反映まで一貫して行う。新機能の追加は `/add-feature` を使う。

**引数:** 修正内容（例: `/fix-code speed-calc の最遅計算で floor の位置がずれている`、または `/review-code` の指摘を貼り付け）

---

## ステップ1: 修正対象と内容の把握

1. 引数の修正内容を分析し、対象ファイル・関数を特定する。
2. `/review-code` の指摘が渡された場合は、指摘を**個別の修正項目**に分解する（複数あれば全項目を後続で消化する）。
3. `Grep` / `Read` ツールで、対象コードと、それに対応する既存テスト（`tests/**`・`tools/PokelensTools.Tests/**`）を把握する。

## ステップ2: プロジェクト理解と規約確認

1. `CLAUDE.md` と `docs/` の関連ドキュメントを確認し、設計思想・アーキテクチャと矛盾しない修正方針を立てる。
2. `Skill('development-guidelines')` のコーディング規約を確認する。

## ステップ3: 修正の実装

1. 特定した修正を `Edit` ツールで適用する。コーディング規約を常に遵守する。
2. 修正項目が複数ある場合は、**すべての項目を消化するまで**繰り返す。理由なく項目をスキップしない。

## ステップ4: テストの追加・更新

修正の性質に応じて、挙動を担保する自動テストを追加・更新する。

- **バグ修正**: 不具合を再現する回帰テストを追加し、修正後に green になることを確認する。
- **リファクタ**: 既存テストが挙動を担保しているか確認する。挙動は変えず、必要なら内部変更に追従してテストを更新する。
- **仕様変更を伴う修正**: 旧挙動のテストを新挙動に合わせて更新する。不要になったテストは削除する。

## ステップ5: 自動テストの実行

1. 以下を順に実行し、全てパスすることを確認する。
  ```bash
  Bash('npm test')
  Bash('npm run lint')
  ```
  C# (`tools/PokelensTools/`) を変更した場合は、加えて次も実行する。
  ```bash
  Bash('dotnet test tools/PokelensTools.Tests')
  ```
2. いずれかが失敗した場合は、原因を分析し、修正コードを適用してから、再度このステップを実行する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

## ステップ6: 実装検証 (サブエージェント起動)

1. `Task`ツールを使用し、`implementation-validator`サブエージェントを起動して品質を検証する。
  - `subagent_type`: "implementation-validator"
  - `description`: "Fix quality validation"
  - `prompt`: "今回修正した `[修正内容の要約]` に関する全ての変更点の品質を検証してください。対象ファイルは `[変更したファイルのパスリスト]` です。コーディング規約、エラーハンドリング、テスト可能性、既存パターンとの整合性、および修正が回帰を生んでいないかを重点的に確認してください。"

**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

## ステップ7: 自動テストケース一覧への反映（テストを追加・変更・削除した場合は必須）

今回のテスト変更を、対応するスナップショット一覧へ反映する。

- フロントエンド（Vitest）・C# 単体テスト → `docs/testing/unit/automated-test-cases.md`
- C# 統合（パイプライン）テスト → `docs/testing/integration/automated-test-cases.md`

**対象ファイルが存在しない場合**は、`Write` ツールで新規作成する。**もう一方の既存一覧（例: `docs/testing/unit/automated-test-cases.md`）の体裁に厳密に倣う**こと。体裁＝ タイトル / 概要 / 他レベルへの相互リンク / テストフレームワーク・対象・実行コマンドの宣言 / 「表の項目」表 / ファイル接頭辞一覧 / 「サマリ」表（ケース数・小計・総合計）/ テストファイル別セクションの表。

反映ルール（`Edit` または上記の新規作成内で適用）:

- 追加したテストは、その一覧の表形式（ケースID `<ファイル接頭辞>-<連番>` / 種別 / テスト観点 / 入力値 / 期待結果 / 備考）に行を追加する。
- 既存テストの観点・入力・期待結果が変わった場合は、該当行を更新する。
- 削除したテストは該当行を削除する。
- サマリ表の「ケース数」「総合計」を更新する。新規テストファイルには新しいファイル接頭辞を割り当て、接頭辞一覧に追記する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ8に進むこと。**

## ステップ8: ブランチ確認（完了報告への付記・非ブロッキング）

**このステップは処理を止めず、自動でブランチを切り替えもしない。完了報告に注意書きを足すだけである。**

1. `Bash('git branch --show-current')` で現在のブランチを確認する。
2. **`main`（デフォルトブランチ）上だった場合のみ**、最終的な完了報告の末尾に次の注意を含める:
  > ⚠️ main 上で修正しました。`/fix-code` はコミットしないため、コミット前に `git switch -c fix/<対象>` を推奨します（未コミットの変更は新ブランチへ移動し、main はクリーンのまま手戻りなく分離できます）。
3. すでに `feature/*` や `fix/*`、`chore/*` などのブランチ上であれば、何も付記しない。

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で自動的に完了となる。

- ステップ3: 指定された修正項目がすべて実装されている。
- ステップ5: `test`, `lint`（C# を変更した場合は `dotnet test`）が全てエラーなく成功する。
- ステップ6: `implementation-validator`サブエージェントの検証をパスする。
- ステップ7: 追加・変更・削除した自動テストが `docs/testing/` の該当一覧に反映されている。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
